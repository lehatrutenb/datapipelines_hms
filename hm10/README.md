# Сравнительный анализ инструментов управления секретами в Kubernetes

## 1. Объекты анализа
1.  **Sealed Secrets** (Bitnami) - *исправление критической уязвимости Native K8s Secrets.*
2.  **HashiCorp Vault** - *стандарт для динамической ротации и строгого аудита.*
3.  **External Secrets Operator (ESO)** - *интеграция с облачными KMS для GitOps.*

> **Важно:** Обычные `Kubernetes Secrets` (native) — **архитектурная дыра**. Хранение `base64` в Git или незашифрованном etcd недопустимо для реальных проектов. Поэтому в сравнении участвует **Sealed Secrets** как минимальный порог безопасности.

## 2. Сравнительная таблица критериев

| Критерий | 🔵 Sealed Secrets (Bitnami) | 🟣 HashiCorp Vault | 🟢 External Secrets Operator (ESO) |
| :--- | :--- | :--- | :--- |
| **Защита Git** | ✅ **Да.** Данные в репозитории — AES-256 шифротекст. | ✅ **Да.** В Git только пути (`secret/db`). | ✅ **Да.** В Git только CRD `ExternalSecret`. |
| **Защита в etcd** | ❌ **Нет.** После расшифровки хранится как `base64`. | ⚠️ **Опционально.** Vault хранит вне кластера. | ❌ **Нет.** Синхронизирует обычный K8s Secret. |
| **Ротация секретов** | ❌ Ручная. Требует пересоздания SealedSecret. | ✅ **Авто.** Динамические секреты БД. | ✅ **Полуавто.** Синхронизация по таймеру из облака. |
| **Локальная разработка** | 🟢 **Просто.** `kubeseal` работает локально. | 🔴 **Сложно.** Нужен локальный Vault или моки. | 🟢 **Просто.** Приложение видит обычный `Secret`. |
| **Отказоустойчивость** | 🟢 Не зависит от внешнего сервера. | ⚠️ Кластер есть (Raft), но сложен. Без KMS — ручной Unseal при сбое. С KMS — авто. | 🟢 Кеширует секреты в кластере. |
| **Сложность внедрения** | 🟢 Низкая (1 контроллер). | 🔴 Высокая (отдельный кластер). | 🟡 Средняя (CRD + настройка IAM). |
| **Аудит (Кто прочитал?)** | ❌ Только логи K8s (без расшифровки данных). | ✅ Полный аудит каждой операции. | ❌ Логи ESO о синхронизации, не о чтении подом. |

## 3. Детализация инструментов

### 🔵 Sealed Secrets (Bitnami)
**Принцип:** Асимметричное шифрование (Pub/Priv Key). Приватный ключ живет только в кластере. Разработчик шифрует секрет публичным ключом **до** отправки в Git.

**Сценарий компрометации:**
- **Git утек:** Злоумышленник видит `encryptedData: AgB8x...` — **бесполезный мусор** без приватного ключа с продакшен-кластера.
- **Кластер взломан:** Злоумышленник делает `kubectl get secret -o yaml` и получает пароль. **Вывод:** Защищает Git/CI, но не защищает рантайм кластера.

**Почему это не "как обычный секрет"?**
- *Native:* Пароль в `base64` путешествует открытым по сети и лежит в Git.
- *Sealed:* Пароль зашифрован на ноутбуке **до** попадания в сеть.

---

### 🟣 HashiCorp Vault
**Принцип:** Централизованное хранилище вне кластера. Приложение получает секрет через Sidecar-контейнер или CSI Driver.

**Ключевая фича: Encryption as a Service (Transit)**
Приложение отправляет plaintext в Vault, Vault возвращает шифротекст. **Мастер-ключ никогда не покидает Vault.**
- *Результат:* Даже если хакер получил дамп БД и доступ к исходникам приложения — данные в БД (`vault:v1:8SDd3...`) бесполезны, так как ключа расшифровки в приложении нет.

**Кластерный режим Vault:**
- **Технология:** Raft Consensus (как в etcd K8s).
- **Нюанс:** После перезагрузки всех нод кластер требует **Unseal** (ввод частей мастер-ключа админами), если не настроен Auto-Unseal через облачный KMS.

---

### 🟢 External Secrets Operator (ESO)
**Принцип:** Мост между Kubernetes и облачным провайдером (AWS Secrets Manager / GCP / Azure Key Vault). Создает обычный K8s Secret, скачивая данные из облака.

**Сценарий компрометации:**
- **Git утек:** Видна ссылка `arn:aws:secretsmanager:...`. Пароля нет.
- **Кластер взломан:** Аналогично Sealed Secrets — пароль в `base64` доступен через `kubectl`.
- **AWS взломан:** Злоумышленник видит оригинал пароля. Защита на уровне CloudTrail (логи аудита AWS).

---

## 4. Почему шифрование etcd не решает проблему Secret'ов?

Включение шифрования etcd (`EncryptionConfiguration`) защищает только от **физической кражи жесткого диска** мастер-ноды.
- **Проблема №1:** Ключ шифрования лежит на той же мастер-ноде. Удаленный взлом кластера = компрометация ключа.
- **Проблема №2:** Ротация ключа в K8s — сложная ручная операция (полный rewrite всех секретов).
- **Проблема №3:** Не защищает от утечки через `kubectl` или Git.

**Вывод:** Шифрование etcd — обязательная гигиена, но не замена Sealed Secrets или ESO.

## 5. Итоговая матрица выбора

| Если ваш кейс... | Рекомендация |
| :--- | :--- |
| Стартап, нужно просто перестать светить пароли в GitHub. | **Sealed Secrets** |
| Микросервисы > 20, нужна частая смена паролей БД на лету без перезапуска. | **HashiCorp Vault** |
| Используете ArgoCD, живете в AWS/GCP/Azure, хотите стандартизировать секреты. | **External Secrets Operator (ESO)** |
| Жесткий Compliance (Банк/PCI DSS). | **HashiCorp Vault + HSM** |
